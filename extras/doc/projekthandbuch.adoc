== Ziel

Der Appserver verwaltet MMORPG-Player und Guilds. Die Anwendung nutzt Bun,
Hono, TypeScript, Prisma, PostgreSQL, Zod, GraphQL Yoga, Keycloak, Pino,
Vitest, Bruno, Docker Compose und k6.

== Architektur

Die Anwendung ist schichtweise aufgebaut:

* `src/app.mts` registriert Middleware, REST-Routen, GraphQL und Error-Handling.
* `src/player/router` enthält REST-Router und Zod-Validierung.
* `src/player/service` enthält die Geschäftslogik und nutzt Prisma.
* `src/player/graphql` enthält Schema, Query- und Mutation-Handler.
* `src/security` prüft Tokens und Rollen aus Keycloak.

== Vorführung

Typische Vorführschritte:

* `bun run tsc`
* `bun run prettier:check`
* `bun run eslint`
* `bun vitest --project unit`
* `docker compose -f extras/compose/player/compose.yml config`
* Appserver mit Backend-Systemen starten
* REST- und GraphQL-Requests in Bruno zeigen
* Lasttest mit `bun run k6` starten

== Wichtige Konzepte

* REST nutzt Ressourcen, HTTP-Methoden, Statuscodes, `ETag` und `If-Match`.
* Zod validiert eingehende DTOs, bevor sie an Prisma weitergereicht werden.
* Prisma bildet Player und Guild auf PostgreSQL-Tabellen ab.
* GraphQL Yoga bietet Query und Mutation als zweite API-Schicht.
* Keycloak liefert Authentifizierung; Rollenprüfung entscheidet über Autorisierung.
* k6 erzeugt Last auf REST- und GraphQL-Endpunkte.
